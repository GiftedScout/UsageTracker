# UsageTracker v0.4.0 版本规划

> **创建日期**: 2026-05-31  
> **状态**: 规划中  
> **前置版本**: v0.3.0  
> **预估发布**: 2026 年 Q4

---

## 概述

v0.4.0 聚焦**使用监管**和**报告增强**两大方向：

1. **监管模式** — 与现有"助手模式"并列，支持超时强制限制 + 当天封禁
2. **详细报告** — 每日时间分布（时段热力图），接近手机端应用时长统计体验

---

## 一、监管模式（Supervisory Mode）

### 1.1 背景与动机

**现状**：当前 UsageTracker 是纯记录工具（"助手模式"），只统计和展示，不干预用户行为。

**新需求**：家长监管、自我约束场景需要**主动限制**功能：
- 超时后提醒 → 警告 → 强制关闭
- 当天再次启动同类型应用 → 直接封禁

### 1.2 模式切换

在 `config.json` 新增字段：

```json
{
  "supervisory_mode": false,
  "assistant_mode": true
}
```

| 模式 | 行为 | 说明 |
|:-----|:-----|:-----|
| **助手模式**（默认） | 仅记录 + 通知提醒 | 当前行为不变 |
| **监管模式** | 记录 + 超时强制限制 | 新增，需要密码/权限才能关闭 |

**网页端设置界面**：通用标签页新增"监管模式"开关，启用后需要设置**管理密码**（防止被监管者自行关闭）。

### 1.3 超时提醒 → 强制关闭流程

每个分类独立配置超时规则：

```json
{
  "categories": [
    {
      "name": "游戏",
      "supervisory": {
        "enabled": true,
        "daily_limit_minutes": 120,
        "grace_period_minutes": 5,
        "enforce_today": true
      }
    }
  ]
}
```

**完整流程**（以"游戏"分类为例，每日限额 120 分钟）：

```
累计使用达 120 分钟
        │
        ▼
  🔔 第一次提醒（Toast 通知）
  "游戏时间已达 120 分钟，请自行关闭"
        │
        ▼ 用户不理睬，启动宽限期定时器（5 分钟）
        │
        ▼ 宽限期结束
  ⚠️ 第二次警告（弹窗 + 声音）
  "5 分钟后将强制关闭游戏进程"
        │
        ▼ 5 分钟后
  💀 强制杀死该分类下所有运行中的进程
  （psutil.terminate() → kill()）
        │
        ▼ 当天再次检测到该分类进程启动
  🚫 立即强制杀死 + 通知
  "今日游戏时间已用完，无法启动"
        │
        ▼ 次日 00:00 自动重置
```

### 1.4 宽限期（Grace Period）设计

| 参数 | 默认值 | 可配置 |
|:-----|:-----|:-----|
| 首次提醒后宽限期 | 5 分钟 | ✅ 网页端可配置 |
| 警告频率 | 每分钟一次 | ✅ 可配置 |
| 强制关闭方式 | terminate() → 等待 3s → kill() | 固定 |

### 1.5 当天封禁逻辑

```
监管模式启用 + enforce_today = true
        │
        ▼ 检测到某分类今日已超时
        │
        ▼ 该分类下任何进程启动时：
  1. 立即 psutil.terminate(process.pid)
  2. 等待 3 秒，若仍存在 → psutil.kill()
  3. 发送 Toast 通知："今日[X]分类已被限制，明天 00:00 重置"
  4. 记录拦截日志到 usage_data.db（新表 supervision_logs）
```

**`supervision_logs` 表结构**：

```sql
CREATE TABLE supervision_logs (
    id          INTEGER PRIMARY KEY,
    timestamp    TEXT    NOT NULL,
    app_name     TEXT    NOT NULL,
    category     TEXT    NOT NULL,
    action       TEXT    NOT NULL,  -- 'warning', 'grace_start', 'force_kill', 'block_today'
    pid          INTEGER,
    note         TEXT
);
```

### 1.6 管理密码

防止被监管者关闭监管模式：

- 启用监管模式时，**必须**设置管理密码（6 位以上）
- 关闭监管模式 / 修改监管规则时需要输入密码
- 密码 Hash 存储在 `config.json`（sha256 + salt）
- **忘记密码**：需要删除配置文件（网页端给出提示）

### 1.7 监管模式网页端设置界面

在网页端设置新增"监管"标签页：

```
┌─ 监管模式 ─────────────────────────────┐
│  ☑ 启用监管模式                        │
│  管理密码：[已设置] [修改] [清除]     │
│                                       │
│  ┌─ 分类规则 ──────────────────┐    │
│  │ 分类    每日限额  宽限期  封禁 │    │
│  │ 游戏    120 分钟  5分钟  ☑   │    │
│  │ 娱乐    60 分钟   3分钟  ☑   │    │
│  │ 社交    180 分钟  10分钟 ☑   │    │
│  └─────────────────────────────┘    │
│                                       │
│  ┌─ 今日拦截记录 ──────────────┐    │
│  │ 14:32  游戏/原神.exe  → 拦截  │    │
│  │ 15:10  游戏/解限机.exe → 拦截  │    │
```

### 1.9 安全与边界情况

| 风险 | 应对 |
|:-----|:-----|
| 被监管者用任务管理器关闭 UsageTracker | 服务化（Windows Service），普通用户无法关闭 |
| 被监管者卸载程序 | 监管密码保护卸载（安装时注册，卸载时验证） |
| 强制杀死导致数据丢失 | 仅对游戏/娱乐分类启用；工作类不启用监管 |
| 误杀系统进程 | 监管只针对用户配置的分类；系统进程白名单 |

## 二、详细报告（时间分布热力图）

### 2.1 背景

**现状**：日报只显示"应用 A：3 小时 20 分钟"，缺乏时间维度信息。

**目标**：类似手机"屏幕使用时间"的**时段分布图**，让用户直观看到"什么时候用了什么"。

### 2.2 每日时间分布图

**数据粒度**：以 **30 分钟**为一个时段（slot），全天 48 个 slot。

**展示形式**：水平柱状图 / 堆叠面积图，X 轴为时间（00:00 ~ 23:30），Y 轴为应用/分类。

```
使用时长分布（2026-05-31）

 游戏  ████████████░░░░░░░░░░░░░░░░░░░░░░░░░░░  120min
 工作  ░░░░░░░░░░████████████████░░░░░░░░  180min
 娱乐  ░░░░░░░░░░░░░░░░░░░░░░░░████████  45min
       00  02  04  06  08  10  12  14  16  18  20  22  24
```

**网页报告（HTML）实现**：用 Chart.js 的 `stacked bar chart`，每个 slot 是一个 bar，不同应用/分类用不同颜色堆叠。

### 2.3 时段聚合数据接口

`src/reporter.py` 新增方法：

```python
def get_time_distribution(self, date: str, group_by: str = "category") -> list[dict]:
    """
    返回指定日期的时间分布数据。
    group_by: "category" | "app"
    返回格式：
    [
      {"time_slot": "09:00", "category": "工作", "minutes": 25},
      {"time_slot": "09:30", "category": "工作", "minutes": 28},
      ...
    ]
    """
```

**SQL 思路**：将 `start_time`/`end_time` 按 30 分钟 slot 聚合。

### 2.4 报告页面改动

**日报 HTML 模板**新增区域（在现有排名下方）：

```html
<!-- 时间分布热力图 -->
<section class="time-distribution">
  <h2>⏱️ 时间分布</h2>
  <div class="slot-legend">
    <!-- 48 个 slot，颜色深浅表示时长 -->
  </div>
  <canvas id="timeDistributionChart"></canvas>
</section>
```

**Chart.js 配置要点**：

```javascript
new Chart(ctx, {
  type: 'bar',
  data: {
    labels: Array.from({length: 48}, (_, i) => `${String(Math.floor(i/2)).padStart(2,'0')}:${i%2*30}`),
    datasets: [
      {label: '工作', data: [...], backgroundColor: '#4CAF50'},
      {label: '游戏', data: [...], backgroundColor: '#F44336'},
      // ...
    ]
  },
  options: {
    scales: {
      x: {stacked: true},
      y: {stacked: true, title: {display: true, text: '分钟'}}
    }
  }
});
```

### 2.5 周报/月报时间分布

- **周报**：叠加 7 天的时间分布，展示"哪天什么时候最活跃"
- **月报**：按周聚合，展示"哪些时段是高频使用时段"

### 2.6 峰值时段识别

报告新增"峰值时段"分析：

```
📊 你的峰值时段：
   上午：09:00 ~ 11:00（平均 85 分钟/天）
   下午：14:00 ~ 16:00（平均 62 分钟/天）
   晚上：20:00 ~ 22:00（平均 110 分钟/天）
```

实现：对时间分布数据按 slot 求 7 天均值，取 Top 3。

### 2.7 涉及文件

| 文件 | 改动 |
|:-----|:-----|
| `src/reporter.py` | 新增 `get_time_distribution()`、`get_peak_slots()` |
| `assets/report_template.html` | 新增时间分布图表区域 + Chart.js 配置 |
| `src/data_store.py` | 新增时间分布聚合查询 SQL |
| `ui/web/` | 网页端报告预览（如果 v0.3.0 已实现） |

## 五、风险与注意事项

| 风险 | 应对措施 |
|:-----|:-----|
| 强制杀死进程导致数据丢失（如未保存的文档） | 仅对"游戏""娱乐"分类默认启用；工作类分类不启用监管；用户可逐分类配置是否启用强制关闭 |
| 监管模式被任务管理器绕过 | v0.4.0 先不做 Service 化，文档说明"需要管理员权限运行"；v0.5.0 考虑 Service 化 |
| 时间分布数据量变大（48 slots × N 天） | 只实时计算，不落盘原始 slot 数据；报告生成时临时聚合 |
| Chart.js 堆叠图在大量数据下渲染慢 | 日报默认显示；周报/月报用聚合视图（按 2 小时合并） |
| 监管密码忘记后无法恢复 | 文档明确说明"忘记密码需手动删除配置文件"；v0.5.0 可考虑密保问题 |
| 玩游戏时强制关闭体验差 | 宽限期设计（默认 5 分钟），给予充分警告；可配置为"仅提醒不强制"模式 |

## 六、关键决策记录

- **监管 vs 助手模式并存**：`config.json` 中两个布尔字段，可独立开关（但同时开启无意义，网页端做互斥提示）
- **强制关闭用 psutil**：跨平台兼容；terminate() 优先，kill() 保底
- **时间分布不落盘原始 slot 数据**：节省存储空间，报告生成时实时聚合（性能可接受，< 500ms）
- **监管日志落盘**：`supervision_logs` 表，用于"今日拦截记录"展示和排错
- **密码 Hash 算法**：SHA-256 + random salt，存储在 `config.json`
